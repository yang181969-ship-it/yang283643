# 性能优化五步收官 —— 从「能看」到「真的快」
@category: 建站日志
@meta: 性能优化
@date: 2026-04-22

今天把拖了好几天的网站性能优化一口气做完了。前几天已经把 ①CSS 压缩和 ②图片 WebP 化搞定,今天从第③步开始一路推到第⑤步,整个站点的加载体验有了肉眼可见的提升。

记录一下完整的五步成果,也算给自己留个存档。

---

## ① CSS 压缩

`package.json` 的 build 脚本加上 `--style=compressed --no-source-map`,Sass 编译产物从几百 KB 砍到 **54.6 KB**。没什么技术含量,但养成"部署用 build 不用 dev"的习惯很重要。

---

## ② 图片 WebP 化 + 智能压缩

这一步是整个优化里收益最高的一环。建了 `scripts/` 目录下三个 Node 脚本组成的流水线:

- `1-rename-anime-images.js` —— 压缩 `assets/animes/` 并按 id 智能重命名成 `{id}-cover.webp` / `{id}-N.webp`
- `2-optimize-gallery.js` —— 压缩 `assets/gallery/`,保留原文件名
- `3-update-html-references.js` —— 读 `rename-mapping.json` 自动更新 HTML 引用,顺便生成 `.bak` 备份

用了 sharp 库,动态质量策略(>500KB 用 92,200-500KB 用 85,100-200KB 用 82,<100KB 用 78),PNG 透明图用无损,webp 反而变大时回退保留原图。

**效果**:gallery 从 5.61MB → 2.44MB,省了 **56.4%**。animes 类似收益。

---

## ③ 图片懒加载

精细控制策略,没搞全站无脑加 `loading="lazy"`:

- `html/anime.html` 的 7 张封面加 `loading="lazy"`(首屏只见得到 2-4 张,剩下的受益)
- `js/anime-detail.js` 的 gallery forEach 里给动态创建的 `<img>` 加 `img.loading = "lazy"`
- logo、lightbox 图片、首屏主图**不加**——这些不该被延迟

`gallery.js` 我自己早就加过了。

---

## ④ PurgeCSS 诊断(停在诊断,不进构建)

跑 PurgeCSS 扫了一圈未使用的 CSS 选择器,结果是**只省了 2.1%(1.2 KB)**。

一开始有点意外,但冷静想想这正说明 Sass 的 ITCSS 7 层架构本身就很精炼。Waline 和 KaTeX 的样式被 safelist 整块保了,确实没什么可删的。

权衡下来,**这一步停在诊断阶段**——脚本和命令留着(`npm run purgecss`)当以后诊断工具用,不进生产构建流程。1.2 KB 的收益不如花时间做第⑤步。

---

## ⑤ 关键 CSS 内联(今天的真正重头戏)

这一步解决的是**白屏时间**——浏览器拿到 HTML 后必须等 CSS 下载解析完才能渲染,慢网下可能有几百毫秒到 1-2 秒。

### 思路

把首屏可见元素的 CSS 直接写进 `<head>` 的 `<style>` 里,剩下的 CSS 异步加载。浏览器拿到 HTML 立刻能渲染,不等网络。

### 实现

新建 `css/scss/critical.scss`,只 `@use` 首屏必需的层:

- `base/theme-vars` + `base/reset`(变量 + 重置)
- `layout/layout` + `layout/header` + `layout/nav`(整体布局)
- `components/hero` + `components/card` + `components/search` + `components/theme-panel` + `components/mode-toggle` + `components/floating-btns`(首屏组件)
- `themes/dark`(必须最后,防闪烁的核心)

故意**不包含** `pages/*`、`components/formula`、`components/lightbox`、`vendor/*`——这些都不在首屏。

新建 `scripts/5-inline-critical.mjs`,自动读 `css/critical.css` 并替换三个 HTML 文件(`index.html` + 两个详情页)的 `<style.css>` 链接,换成内联 `<style>` + 异步 `<link rel="stylesheet" media="print" onload="this.media='all'">`。

### 踩的坑

第一次跑脚本只成功了两个文件。原因是我脚本的正则写的是 `href="./css/style.css"`(带 `./`),但 `index.html` 里是裸路径 `href="css/style.css"`,没匹配上。加了第三条正则就好了。

### 效果

**首页刷新后首屏几乎瞬间出现**,主题切换零闪烁。暗色模式下连着刷新几次也没有"从白到深"的瞬变——因为 `[data-theme="dark"]` 的核心背景色已经内联在 HTML 里,和 `theme.js` 同步到位。

---

## 构建流程固化

以后改 CSS 的标准流程变成三步:

```bash
npm run build            # 编译完整 style.css
npm run build:critical   # 编译 critical.css
npm run inline:critical  # 内联进三个 HTML
```

加了个组合命令 `npm run build:all` 一键搞定。

## 总结

| 步骤 | 措施 | 核心收益 |
|------|------|---------|
| ① | CSS 压缩 | 产物 → 54.6 KB |
| ② | 图片 WebP + 智能压缩 | 省 50%+ 体积 |
| ③ | 图片 `loading="lazy"` | 首屏请求减少 |
| ④ | PurgeCSS 诊断 | 证实源码已精炼(2.1% 冗余) |
| ⑤ | 关键 CSS 内联 + 异步 | **消除白屏与主题闪烁** |

## 一点题外话

做完第⑤步的瞬间有点意外——本来以为白屏这种问题得改架构才能解决,结果就是"把首屏那点样式挪到 HTML 里"这么朴素的思路。工程优化很多时候就是这样,原理一点都不玄,难的是规划清楚边界(哪些该内联、哪些不能内联)和踩过每一个小坑(路径正则、Windows 下 Node 24 的 ESM 兼容问题、.bak 备份的回滚策略...)。
